# Массовая загрузка товаров и фото

## Логика

Одна строка в шаблоне = один основной артикул / один принт. Сайт сам создает варианты по столбцам:

- `Основной артикул`: для папок `opt_67895` пишем `opt_67895`, без удаления префикса `opt_`
- `Типы товара`: например `Подушка; Наволочка`
- `Категории`: например `Подушки; Наволочки`
- `Размеры`: например `30x30; 35x35; 40x40; 45x45; 50x50`
- `Материалы`: например `Велюр; Габардин`
- `Статус публикации`: `draft`, `published`, `hidden` или `archive`. Новые строки в шаблонах по умолчанию идут как `draft`, чтобы партия не появилась в публичном каталоге до проверки.
- `Статус наличия`: например `made` или `ready`. Старую колонку `Статус` импортер по-прежнему читает как статус наличия.

В CSV-файлах разделитель колонок тоже `;`. Внутри ячеек со списками несколько значений также разделяются через `;`, поэтому Excel будет брать такие ячейки в кавычки автоматически. Старые файлы с запятыми в ячейках тоже читаются, если в ячейке нет `;`, но новые файлы лучше вести через `;`, чтобы размеры вроде `3,5` не ломались.

Правило артикула варианта: `Основной артикул_ТИП_Размер_МАТ`, где разделитель `_`, тип товара и материал сокращаются до первых 3 символов после очистки, а размер пишется полностью. Примеры: `Подушка` -> `ПОД`, `Наволочка` -> `НАВ`, `Велюр` -> `ВЕЛ`, `Габардин` -> `ГАБ`, `Oxford` -> `OXF`.

Если один принт должен попадать в две категории, в колонке `Категории` пишем их через `;`: `Подушки; Наволочки`. В колонке `Типы товара` для такого товара можно написать `Подушка; Наволочка`. Если название базовой карточки начинается с `Подушка`, то вариант с типом `Наволочка` на сайте будет отображаться как `Наволочка ...` автоматически.

Каждая комбинация `тип × размер × материал` считается отдельным артикулом варианта. В админке можно скачать отдельную таблицу цен вариантов: там каждая строка = один вариант с колонками `Основной артикул`, `Артикул варианта`, `Тип товара`, `Размер`, `Материал`, `Цена варианта`. Это нужно для редактирования цены каждого варианта отдельно. Для текущей партии локально также формируется файл `local-import-output/variant-prices.xlsx`.

Фото ищутся по папке основного артикула. Пример:

```text
Фото товаров/
  10345/
    1.jpg
    2.jpg
    3.jpg
  10346/
    1.png
    2.webp
```

Также поддерживается вложенная структура `категория / артикул / фото`. В этом случае сканер берет категорию из предпоследней папки, основной артикул из последней папки, приводит его к виду `opt_число`, а относительный путь записывает в колонку `Папка фото`.

```text
Фото товаров/
  пляжные сумки/
    opt_Акулы_68029/
      1.jpg
      2.jpg
  шевроны 8х10/
    48100/
      1.jpg
```

`1.jpg` обычно становится главным фото, остальные уходят в галерею. Лучше держать фото квадратными `1200x1200 px`, JPG/WebP.

## Если список артикулов уже есть

Скачать шаблон можно из админки кнопкой `XLSX-шаблон` или из файла:

```text
templates/sobag-products-template.xlsx
```

После заполнения можно загрузить таблицу через админку или прогнать локальный импортер.

## Если списка артикулов нет

Импортер умеет собрать черновой шаблон из названий папок с фото:

```powershell
python tools/product_importer.py scan-photos --photos "C:\Path\YandexDisk\Фото товаров" --out local-import-output\products-from-photo-folders.xlsx
```

После этого файл `products-from-photo-folders.xlsx` можно открыть, дописать названия, категории, типы товара, подборки, праздники, цены и загрузить обратно.

Если поле `Подборки` пустое, товар не добавляется ни в одну подборку. Новые категории из таблицы добавляются на сайт автоматически при импорте через админку; временную иконку и описание можно поменять в админке позже.

При загрузке новых товаров старые товары не удаляются. Система проверяет `Основной артикул`: если такой артикул уже есть, строка считается дублем и пропускается. Повторная загрузка того же файла не должна создавать копии товаров. Публичный каталог показывает только товары со статусом публикации `published`; `draft`, `hidden` и `archive` остаются доступными в админке и экспортах.

## Партии импорта в админке

`admin-import.html` теперь работает через партии импорта. Загрузка Excel/CSV сначала создает preview-партию и не меняет каталог до нажатия `Применить`.

Перед загрузкой файла можно включить режим `Обновлять существующие товары по baseSku`. Без него строки с уже существующим `baseSku` попадают в отчет как `duplicate_skipped`; с ним такие строки получают action `updated`. Импорт не удаляет старые товары, а при обновлении сохраняет текущие `status`, `image`, `gallery` и `images`, если соответствующие колонки/metadata не переданы в файле.

В карточке партии видны:

- статус партии: `preview`, `applied`, `rejected`, `rolled_back`;
- счетчики `created`, `skipped`, `updated`, `errors`;
- построчный отчет с причиной пропуска или ошибки;
- кнопки `Применить`, `Отклонить`, `Скачать отчет CSV`;
- `Откатить последнюю партию` для последней примененной партии со snapshot.

Правила безопасности:

- импорт не удаляет существующие товары;
- дубли по `baseSku` пропускаются;
- коллизии рассчитанных SKU вариантов пропускаются;
- preview можно отклонить без изменений каталога;
- apply сохраняет snapshot каталога перед записью;
- rollback доступен только для последней примененной партии нового механизма.

Если Vercel API недоступен локально, страница использует localStorage fallback для preview/report, но production-сценарий должен идти через `/api/admin/import-batches`.

## Фото и object storage

Новые bulk-фото больше не должны попадать в Git. Для следующих партий используем object storage слой из `api/_lib/object-storage.js` и документацию `docs/object-storage.md`.

Первый provider: Vercel Blob. В репозитории хранится только код и metadata, а секрет `BLOB_READ_WRITE_TOKEN` должен жить только в окружении Vercel/локальной машине. В товаре можно хранить старые поля `image`/`gallery` для совместимости и новое поле `images` с metadata: `url`, `storageKey`, `provider`, `width`, `height`, `mime`, `uploadedAt`.

В `admin-import.html` после загрузки Excel/CSV можно выбрать файлы или папку фото. Страница сопоставляет изображения с товарами по `baseSku` и `Папка фото`, показывает отчет `ready/missing/repeated/uploaded/failed`, умеет скачать CSV-отчет и после успешной загрузки создает новый preview batch с Blob/S3 metadata.

Для больших партий используйте CLI, чтобы не передавать файлы через JSON API:

```powershell
node tools/bulk-upload-product-photos.mjs --products data/products.import.json --photos "C:\Path\Photos" --dry-run
node tools/bulk-upload-product-photos.mjs --products data/products.import.json --photos "C:\Path\Photos" --out local-import-output\products-with-object-images.json --report local-import-output\bulk-photo-upload-report.csv --responsive
```

Реальный upload требует `BLOB_READ_WRITE_TOKEN` только в локальном/production окружении. Для реальной генерации responsive WebP/AVIF variants нужен optional `sharp`; dry-run строит план variants без него. CLI пишет CSV-отчет и products JSON с `images` metadata, не добавляя bulk-фото в Git.

## Локальный импорт

```powershell
python tools/product_importer.py import --table local-import-output\products-from-photo-folders.xlsx --photos "C:\Path\YandexDisk\Фото товаров"
```

По умолчанию локальный импортер сохраняет старые товары в `data/products.import.json` и добавляет только новые артикулы. Если `data/products.import.json` еще не создан, он берет базу из `data/products-live.json`, чтобы случайно не опубликовать только новую маленькую партию вместо всего каталога.

Если в таблице написан чистый числовой артикул, например `67895`, импортер сам приводит его к виду `opt_67895`. Повторная загрузка того же файла не должна создавать копии товаров: дубли по основному артикулу и дубли рассчитанных артикулов вариантов пропускаются и записываются в отчет. При `--update-existing` существующий статус публикации сохраняется, если в таблице нет колонки `Статус публикации`.

После импорта создаются:

- `data/products.import.json` - полный черновой каталог: старые товары + новые товары;
- `data/import-report.csv` - отчет со статусами `created`, `duplicate_skipped`, `variant_duplicate_skipped`;
- `local-import-output/variant-prices.xlsx` и `local-import-output/variant-prices.csv` - таблица цен всех рассчитанных вариантов.

Полностью заменить выходной JSON можно только явной командой с флагом `--replace`.

Результат:

- `assets/imported-products/` - скопированные фото;
- `data/products.import.json` - подготовленные товары для будущего подключения к сайту/backend;
- `data/import-report.csv` - отчет по строкам и фото.

Эти выходные папки сейчас добавлены в `.gitignore`, чтобы случайно не загрузить тысячи фото в GitHub.

## Публикация импортированной партии на сайт

После проверки `data/products.import.json` и отчета можно подготовить live-данные и оптимизированные WebP:

```powershell
python tools/publish_imported_products.py --input data/products.import.json --out data/products-live.json --source-assets assets/imported-products --target-assets assets/product-preview-live
npm run check
```

Публикация теперь собирает изображения во временную папку и заменяет целевую папку только после успешной подготовки. Это снижает риск потерять уже оптимизированные изображения, если сборка оборвется. Фото вне `assets/imported-products` не трогаются: например fallback-изображения остаются как есть.

Для флагов порядок фото сохраняется `1, 2, 3`. Для остальных категорий главное фото выбирается с конца списка, как мы договорились после прошлой загрузки. Категория считается флаговой, если в названии категории есть корень `флаг`.

## Яндекс Диск

Самый удобный вариант: установить приложение Яндекс Диска и синхронизировать папку с фото на компьютер. Тогда для импортера это обычная локальная папка, например:

```powershell
--photos "C:\Users\SoBag\YandexDisk\Фото товаров"
```

Давать пароли или токены в чат не нужно. Если позже понадобится импорт напрямую по ссылкам/API, лучше подключать это отдельным безопасным способом.

## Проверка перед публикацией

Сейчас `data/products-live.json` хранит базовые карточки: один основной артикул = один принт/товарная карточка. Варианты для сайта генерируются автоматически из `Типы товара`, `Размеры` и `Материалы`. Поэтому в самом JSON массив `variants` может отсутствовать, а в админке и карточке товара варианты все равно будут рассчитаны.

Перед публикацией большой партии нужно запускать:

```powershell
npm run check
```

Эта проверка теперь дополнительно контролирует структуру товаров:

- основной артикул не пустой и не повторяется;
- обязательные списки `categories`, `types`, `sizes`, `materials` заполнены;
- внутри списков нет дублей с разным регистром;
- все фото существуют на диске;
- тестовые изображения и старые `SB-...` артикулы не попали обратно;
- все рассчитанные артикулы вариантов уникальны;
- цены вариантов получаются положительными.
- если указан `status`, он входит в список `draft/published/hidden/archive`, а старый `hidden` согласован с ним.

Для быстрой проверки интерфейса можно запустить локальный сервер и smoke-тест:

```powershell
python -m http.server 4173 --bind 127.0.0.1
$env:SOBAG_BASE_URL='http://127.0.0.1:4173'
npm run ui:smoke
```

Локальный Python-сервер не поднимает `/api`, поэтому smoke-тест специально игнорирует локальные 404 на API и проверяет статическую витрину: каталог, избранное, фильтры, карточку, смену варианта, добавление в корзину и поиск.
